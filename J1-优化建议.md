# J1 — Java 端 McpController 优化建议

## 优化项 1：Bindings 返回格式统一为 Array

### 问题
当前 `getBindings()` 返回 `Map<String,String>` 对象格式：
```json
{"UWB001": "张三", "UWB002": "李四"}
```
TypeScript 端需要额外写兼容逻辑来处理 object 和 array 两种格式。

### 建议修改
改为返回统一数组格式，与 TypeScript 端 `TagBindingEntry[]` 对齐：
```java
@GetMapping("/realtime/bindings")
public GlobalResponse<List<Map<String, Object>>> getBindings() {
    List<Map<String, Object>> result = new ArrayList<>();
    for (Map.Entry<String, String> entry : gisDataService.tagCodeBindMap.entrySet()) {
        Map<String, Object> item = new LinkedHashMap<>();
        item.put("tagCode", entry.getKey());
        item.put("bindName", entry.getValue());
        item.put("bindType", "personnel");  // 可根据实际业务扩展
        item.put("bindId", 0);
        result.add(item);
    }
    return new GlobalResponse(result);
}
```

### 影响
- TypeScript 端 `callTagBindings()` 可移除 object→array 兼容分支
- 返回格式与 `getTagBindings` 工具完全一致

## 优化项 2：GlobalResponse 统一状态码

### 问题
当前 `GlobalResponse` 成功时只有 `result` 字段，失败时只有 `message` 字段：
- 成功：`{"result": true}`
- 失败：`{"message": "绑定失败: xxx"}`

TypeScript 端 `callMcpWrite` 期望的格式是 `{code, message, result}`，需要判断 `res.result === true || res.code === 200` 两种成功模式。

### 建议修改
让 `GlobalResponse` 统一返回 3 个字段（需修改 `GlobalResponse` 类或在 Controller 层包装）：

**方案 A：修改 GlobalResponse 类**（推荐，但影响面大）
```java
@Data
public class GlobalResponse<T> {
    private int code;         // 200=成功, 其他=错误码
    private String message;   // 消息
    private T result;         // 数据

    // 成功构造函数
    public GlobalResponse(T result) {
        this.code = 200;
        this.message = "success";
        this.result = result;
    }

    // 失败构造函数
    public GlobalResponse(MessageEnum enumMsg, String detail) {
        this.code = enumMsg.getCode();
        this.message = detail;
        this.result = null;
    }
}
```

**方案 B：在 Controller 层保持兼容**（最小改动）
如果 `GlobalResponse` 被其他模块广泛使用不能改，则在 Controller 中保持一致即可，TypeScript 端当前已有兼容逻辑。

### 影响
- TypeScript 端可简化成功判断逻辑
- 错误处理更精确

## 优化项 3：读操作可选鉴权

### 问题
当前读操作（`/realtime/*`）无任何鉴权，任何知道 URL 的人都能查询数据。

### 建议修改
加可选的 API Key 校验——传了 Key 就验证，不传也放行（兼容现有调用）：

```java
/**
 * 可选 API Key 校验（读操作）
 * 传了 Key 就验证，不传则放行（兼容现有调用）
 */
private void checkReadApiKey(@RequestParam(required = false) String apiKey) {
    if (apiKey == null || apiKey.isEmpty()) {
        return; // 不传 Key 也放行
    }
    String expected = environment.getProperty("mcp.api.key");
    if (expected == null || expected.isEmpty()) {
        return; // 服务端没配置 Key，放行
    }
    if (!expected.equals(apiKey)) {
        throw new CustomException(Collections.singletonList("Invalid MCP API Key"), MessageEnum.FAILED);
    }
}
```

在需要的读接口上加 `@RequestParam(required = false) String apiKey` 参数并调用校验。

### 影响
- 提升安全性而不破坏现有调用
- 敏感查询（人员位置、进出记录）建议加鉴权

## 优先级建议

| 优先级 | 项目 | 工作量 | 收益 |
|--------|------|--------|------|
| P1 | 优化项 1（Bindings 格式） | 15 分钟 | 中（简化 TS 端代码） |
| P2 | 优化项 2（状态码统一） | 1-2 小时 | 高（需评估 GlobalResponse 影响面） |
| P3 | 优化项 3（可选鉴权） | 30 分钟 | 中（提升安全性） |

> **建议**: 先做 P1（改动最小，收益明确），然后评估 P2 的影响面，最后加 P3。
